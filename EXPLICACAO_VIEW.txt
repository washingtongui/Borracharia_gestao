Por que a view foi alterada para usar conexão direta (pyodbc)

Resumo:
A busca de clientes foi transferida do ORM do Django para uma consulta direta usando pyodbc. Isso foi necessário porque, com o backend atual do SQL Server (sql_server.pyodbc) e a versão do Django em uso, determinadas operações do ORM geravam uma exceção interna (django.core.exceptions.FullResultSet) durante a compilação da cláusula WHERE, impedindo a execução da consulta e quebrando a renderização do template.

Causas principais identificadas:
- Incompatibilidade entre o backend `sql_server.pyodbc` (ou suas adaptações) e a versão do Django (6.x), que leva a falhas ao compilar certas expressões SQL.
- Diferenças no esquema/nomes de tabelas (caixa/DB table name) e comportamentos do driver ODBC que podem tornar consultas ORM mais frágeis.
- Problemas de permissão no banco (SELECT negado) que foram detectados antes, e posteriormente resolvidos ao trocar o usuário.

Por que usar pyodbc direto (vantagens imediatas):
- Robustez: ao executar SQL direto, evitamos a camada que estava quebrando (o compilador SQL do backend) e conseguimos retornar dados ao template.
- Controle: podemos escrever SQL explícito, tratar erros e mapear resultados de forma previsível.
- Diagnóstico: facilita isolar problemas de driver/permissão versus problemas de ORM/DSL.

Trade-offs e desvantagens:
- Perda de abstração: deixar de usar o ORM significa abrir mão de conveniências como conversão automática de tipos, relacionamentos e validações do modelo.
- Manutenção: queries escritas à mão exigem atenção para mudanças de esquema e podem repetir lógica presente nos models.
- Segurança/SQL injection: é preciso garantir que parâmetros usados em SQL sejam passados corretamente via placeholders (pyodbc) — a implementação atual usa strings estáticas e não recebe parâmetros do usuário, então o risco é baixo, mas atenção é necessária para queries dinâmicas.

Próximos passos recomendados (se você preferir voltar a usar o ORM):
1. Atualizar/verificar o backend SQL Server compatível com Django 6 (por exemplo, verificar versões suportadas do `django-mssql-backend` ou alternativas) e o driver ODBC (ODBC Driver 17/18).
2. Testar com uma instalação mínima/isolada reproduzindo a operação que levanta `FullResultSet` e abrir issue com o mantenedor do backend se necessário.
3. Verificar nomes de tabelas/colunas (db_table) e schema (ex.: dbo.Clientes) para garantir mapeamento correto no `models.py`.
4. Garantir permissões corretas do usuário na base (`GRANT SELECT` ou adicionar ao role `db_datareader`).
5. Se o ORM for essencial, considerar escrever um pequeno wrapper que capture exceções específicas do compilador e faça fallback para SQL direto apenas nesses casos.

Conclusão:
A mudança para pyodbc direto foi uma solução pragmática e segura para restaurar a funcionalidade da página rapidamente, garantindo que os usuários vejam os dados. É temporária do ponto de vista da arquitetura: o ideal é corrigir a incompatibilidade entre o backend do SQL Server e o ORM, mas isso requer atualizações e testes na cadeia de dependências (driver, backend, Django). Se preferir, posso: (a) tentar restaurar o uso do ORM investigando versões/patches; (b) refatorar a consulta direta para parametrização e paginação; ou (c) melhorar o template/apresentação. Diga qual opção prefere.
